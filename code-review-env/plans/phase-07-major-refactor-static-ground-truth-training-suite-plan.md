# Phase 07 Plan - Major Refactor to Deterministic Ground Truth + Training Suite (for GPT-5.3)

## 1) Mission
Refactor GraphReview from LLM-first bug detection into a deterministic-analysis-first RL environment where:
- Ground truth is generated by static analyzers and stored in SQLite.
- Agent reasoning is evaluated against that ground truth.
- Explanations are generated after grading and do not affect deterministic reward.

This plan is for implementation by GPT-5.3. It is task-based, repository-aware, and designed to avoid breaking prior completed phases.

## 2) Current Repository Baseline (As-Is)
The repository already implements through Phase 6 with:
- OpenEnv runtime and API endpoints.
- SQLite-backed graph and seeded findings.
- Deterministic easy/medium graders and hybrid hard grader.
- Edge summaries and optional LoRA hooks.

Major refactor objective:
- Replace "LLM as detector/judge" assumptions for medium/hard truth generation.
- Introduce deterministic truth stack: mypy + pyright + semgrep (+ existing pylint/bandit/pyflakes, optional vulture).
- Keep OpenEnv contract and DB-first architecture intact.

## 3) Critical Design Questions To Resolve Before Coding
GPT-5.3 must ask and resolve these before touching schema or interfaces:
1. Should existing findings table be extended with analyzer metadata, or should a new normalized table store all analyzer outputs?
2. Should pyright be required in all environments or optional with capability flag?
3. Should semgrep custom rules live in repository under a versioned rules directory, and should rule version be persisted per finding?
4. Should training runs write to separate run tables or reuse existing episode/task tables with a run_type discriminator?
5. Should sample_project replacement be destructive (exact file set only) or additive via a new canonical fixture folder?

Do not start implementation until these are confirmed.

## 4) Context7-Validated Guardrails (Must Follow)
From current docs lookup:
- OpenEnv: environment must implement reset, step, and state lifecycle with typed models.
- SQLAlchemy ORM 2.0: use explicit session boundaries and transaction-scoped writes for deterministic persistence.
- Pydantic v2: enforce strict models with forbidden extra fields and explicit validation entry points.

## 5) Target Architecture After Refactor
### 5.1 Detection and Ground Truth Layer (Deterministic)
Truth sources:
- pylint
- pyflakes
- bandit
- mypy
- pyright
- semgrep (custom cascade rules)
- vulture (advisory or low-weight signal)

Output:
- Unified finding records in SQLite with analyzer, rule_id, severity, file, line, evidence, and run_id.
- Ground truth snapshots are immutable per seed run.

### 5.2 Agent Layer (Reasoning, Not Detection)
Agent model:
- Qwen2.5-Coder-7B-Instruct GGUF from Models folder.

Agent task:
- Review code and graph context without being given deterministic findings.
- Propose issues, root-cause attributions, and confidence.

### 5.3 Grader Layer (Deterministic)
- Compare agent outputs against stored deterministic findings.
- Reward/punish based on precision, recall, attribution correctness, and graph consistency.
- No explanation scoring in deterministic reward path.

### 5.4 Explanation Layer (Non-Deterministic but Non-Scoring)
Model:
- Gemma 4 for summarization and edge relationship narrative.

Role:
- Generate post-hoc explanations only after deterministic grading is complete.
- Explanation quality does not modify reward.

## 6) Refactor Phases (Do One Phase At A Time)

## Phase A - Freeze, Baseline, and Compatibility Harness
Objective:
- Lock current behavior and create regression baseline before refactor.

Tasks:
1. Capture current passing test matrix and artifact outputs.
2. Add baseline snapshot tests for reset/step/state and existing report generation.
3. Add migration safety checks that fail if prior interfaces are accidentally broken.

Exit criteria:
- Baseline tests green.
- Snapshot artifacts preserved for comparison.

## Phase B - Ground Truth Schema and Storage Refactor
Objective:
- Create deterministic analyzer storage model without losing historical data.

Tasks:
1. Add normalized analyzer findings table(s), for example:
   - analyzer_runs
   - analyzer_findings
   - analyzer_rule_catalog (optional)
2. Persist per-run metadata:
   - analyzer name/version
   - command/config hash
   - started_at/finished_at
3. Add migration scripts and backward-compatible read adapters.
4. Keep DB as single source of truth.

Exit criteria:
- New schema migrated successfully.
- Old consumers still work via adapters or compatibility views.

## Phase C - Deterministic Analyzer Pipeline Integration
Objective:
- Build a single pipeline that executes all deterministic analyzers and stores unified findings.

Tasks:
1. Add analyzer runner modules under a dedicated package, for example:
   - analyzers/pylint_runner.py
   - analyzers/pyflakes_runner.py
   - analyzers/bandit_runner.py
   - analyzers/mypy_runner.py
   - analyzers/pyright_runner.py
   - analyzers/semgrep_runner.py
   - analyzers/vulture_runner.py
2. Add a normalizer converting tool outputs into one strict Pydantic finding schema.
3. Persist all outputs in one transaction per analyzer run.
4. Add deterministic deduplication strategy.
5. Add timeout and failure handling with explicit status.

Exit criteria:
- One seed invocation produces unified findings from all configured analyzers.
- Repeated runs on unchanged code produce identical normalized findings (except timestamps/run ids).

## Phase D - Semgrep Custom Rule System for Cascades
Objective:
- Encode project-specific cascade bugs as deterministic semgrep rules.

Tasks:
1. Create semgrep rules directory, versioned in repo.
2. Implement initial rules for:
   - None return not checked after call.
   - Config key missing propagation patterns.
   - Unsafe dataflow/cascade patterns specific to sample project.
3. Persist semgrep rule id/version in findings.
4. Add rule unit tests with positive/negative fixtures.

Exit criteria:
- Cascade bugs are reproducibly detected by semgrep rules.
- Rule version is traceable in DB and reports.

## Phase E - Sample Project Canonicalization
Objective:
- Align sample project with required challenge specification.

Required target files:
- auth.py
- checkout.py
- cart.py
- config.py
- database.py
- utils.py
- models.py
- payments.py
- api.py
- main.py

Tasks:
1. Reconcile existing sample_project content with canonical list.
2. Ensure injected bugs match requested mapping exactly.
3. Add a fixture validator script that verifies file set and injected-pattern signatures.
4. Keep at least one clean module path (models.py) for APPROVE tests.

Exit criteria:
- Validator confirms canonical sample project layout and bug fixtures.
- Task mapping is deterministic and documented.

## Phase F - Agent Prompting and No-Leak Evaluation Loop
Objective:
- Ensure the agent is not given answers and must reason from code plus graph context.

Tasks:
1. Redesign hard issue finder and agent inference prompts:
   - Multi-paragraph, explicit reasoning objectives.
   - No deterministic findings exposed in prompt payload.
2. Add prompt hash persistence for reproducibility.
3. Add anti-leak checks ensuring analyzer findings are not passed to agent context.
4. Add configurable action schema for issue proposals:
   - suspected issue
   - evidence snippets
   - attributed root module
   - confidence score

Exit criteria:
- Agent receives only allowed context.
- Prompt and context are reproducible and auditable.

## Phase G - Deterministic Reward and Grader Overhaul
Objective:
- Grade against deterministic findings and graph truth only.

Tasks:
1. Easy grader:
   - Use pylint/pyflakes/bandit findings.
2. Medium grader:
   - Use mypy + pyright findings (None handling, type flow errors).
3. Hard grader:
   - Use semgrep custom rules + graph path validity checks.
4. Add strict attribution scoring:
   - correct bug
   - correct file/line neighborhood
   - correct upstream root cause when required
5. Keep deterministic behavior for easy and medium.
6. Keep hard scoring deterministic except optional explanation generation, which is non-scoring.

Exit criteria:
- Same input sequence yields same easy/medium cumulative reward.
- Hard scoring reproducible with deterministic gates and rule matching.

## Phase H - Training Suite for Qwen2.5-Coder GGUF
Objective:
- Build a robust training/evaluation suite around environment feedback.

Tasks:
1. Add training run orchestration:
   - run episodes
   - collect trajectories
   - compare agent predictions vs deterministic findings
2. Add dataset builder for preference/SFT/RL style records:
   - prompt
   - action
   - deterministic outcome
   - reward
   - correction notes
3. Add configurable reward shaping reports:
   - false positives
   - false negatives
   - attribution misses
4. Add reproducible run manifests with seed, model, analyzer versions.
5. Keep training optional and isolated from core environment runtime.

Exit criteria:
- Reproducible training artifacts generated from environment runs.
- Comparative metrics exported per run.

## Phase I - Gemma 4 Explanation and Critical Analysis Pipeline
Objective:
- Use Gemma 4 exclusively for explanation, summarization, and critical analysis text.

Tasks:
1. Add post-grading summarizer module that consumes:
   - deterministic findings
   - agent findings
   - mismatch analysis
2. Generate outputs:
   - edge relationship text summaries
   - what agent found correctly
   - what agent missed or misattributed
   - suggested next focus areas
3. Ensure these outputs are clearly labeled non-scoring artifacts.

Exit criteria:
- Reports include deterministic score and separate explanatory narrative.
- No reward dependency on summarizer output.

## 8) Strict Contracts To Preserve
1. OpenEnv reset/step/state API behavior remains valid.
2. Reset does not re-parse codebase; it clears run-scoped annotations only.
3. Token budget remains a hard maximum of 2000 tokens per observation.
4. DB remains source of truth for findings, annotations, and episode state.
5. inference.py logging format contract remains intact where required by challenge gate.

## 9) Verification Matrix (Mandatory)

## 9.1 Determinism Tests
1. Easy grader replay x10 yields identical rewards.
2. Medium grader replay x10 yields identical rewards.
3. Analyzer pipeline on unchanged code yields stable normalized findings.

## 9.2 Correctness Tests
1. Mypy/pyright detect None-risk medium fixture as expected.
2. Semgrep custom rules detect hard cascade fixture as expected.
3. Graph attribution checks validate edge/path for hard findings.

## 9.3 Non-Leak Tests
1. Prompt payload scanner verifies no deterministic findings leakage to agent.
2. Agent outputs differ by reasoning, not by pre-exposed answers.

## 9.4 Reporting Tests
1. Deterministic score and explanatory narrative are separated.
2. Gemma 4 outputs exist even when not used in scoring.
3. Critical analysis section lists agent misses and false positives.

## 10) Security and Quality Gates
Run on each phase completion:
- ruff
- mypy (strict on env/graders/analyzers/llm)
- pytest
- bandit
- semgrep (security + custom rules)
- pip-audit

Advisory:
- vulture for dead code signal

## 11) Rollout Strategy (Low-Risk)
1. Feature-flag new analyzer pipeline while retaining old path temporarily.
2. Switch graders to new findings backend once parity checks pass.
3. Remove legacy LLM-detection dependencies after deterministic pipeline is verified.
4. Keep migration scripts idempotent and reversible where possible.

## 12) Final Definition of Done
The refactor is complete when:
1. Deterministic analyzers (including mypy, pyright, semgrep) generate and persist ground truth in SQLite.
2. Agent (Qwen2.5-Coder GGUF) reviews code without receiving precomputed answers.
3. Graders score against deterministic findings and graph truth with reproducible outcomes.
4. Gemma 4 generates explanation and critical analysis as non-scoring artifacts.
5. Sample project exactly matches required canonical fixture and task mapping.
6. OpenEnv lifecycle, token budget, and DB-first guarantees remain intact.
7. Full verification matrix passes and reports are reproducible.
